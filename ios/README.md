# iOS thin renderer — Adaptive Registration Form (Proof of Concept, POC)

SwiftUI shell implementing a hybrid rendering approach (see [`plan.md`](../plan.md) for the full
design): a step-renderer **registry** that dispatches each server-declared `next_step.type` to
either a native SwiftUI screen or a `WKWebView` pointed at the web renderer. The app never encodes
the flow itself — it only knows how to render the six step types (`camera`, `signature`, `pin`,
`form`, `document`, `external` — see `docs/contract.md` for the full API spec) and how to ask
"what's next?".

## Setup

### Option A — regenerate with XcodeGen

```sh
brew install xcodegen        # if you don't have it
cd ios
xcodegen generate
open AdaptiveRegistrationForm.xcodeproj
```

Scheme: **AdaptiveRegistrationForm**. Pick any iPhone simulator (iOS 16+) and Run.

### Option B — open the already-generated project

`AdaptiveRegistrationForm.xcodeproj` is committed as-is (generated by the command above), so you
can also just open it directly without XcodeGen installed. If you add/remove/rename any file
under `App/Sources/`, re-run `xcodegen generate` (or add the file to the Xcode project manually)
— the `.xcodeproj` is a generated artifact, `project.yml` + `App/Sources/**` are the source of
truth.

### Backend

Point the app at your Go backend via **Settings → Backend → Base URL** (gear icon, top-right).
Defaults to `http://localhost:8080`. The same host also serves the web renderer at `/web/`
(WKWebView loads `{baseURL}/web/#/step/{stepId}?session={sessionId}`).

`NSAppTransportSecurity.NSAllowsArbitraryLoads` is set in `project.yml` to allow plain HTTP to
localhost — POC only; a real build pins HTTPS and removes this (flagged inline in `project.yml`).

```
    │   ├── StepDefinition.swift   #   next_step (all types) + FormField
    │   └── SessionModels.swift    #   session start/resume, submit bodies, upload slot, errors, refdata
    ├── Networking/
    │   ├── APIClient.swift        # thin async/await REST client (session start/resume, step submit, refresh, uploads)
    │   └── APIError.swift
    ├── ViewModels/
    │   └── SessionViewModel.swift # the "thin shell" orchestrator: bootstrap/resume/submit/refresh
    ├── Registry/

    ├── Steps/                     # per-type renderers
    │   ├── CameraStepView.swift       # native: camera/PhotosPicker/mock-capture + Vision face check
    │   ├── SignatureStepView.swift    # native: Canvas + drag gesture → PNG upload
    │   ├── PinStepView.swift          # native: 6-digit pad, enter+confirm, never persisted
    │   ├── NativeFormStepView.swift   # native comparison renderer (Settings toggle; text/select/bool/date only)
    │   └── WebStepView.swift          # WKWebView renderer — form/document (always) + external escape hatch
    ├── Views/                     # shell chrome around whatever the registry renders
    │   ├── RootView.swift, ProgressBarView.swift, BannerView.swift (banners + repairs)
    │   ├── MaintenanceView.swift, CompletionView.swift, SettingsView.swift
    └── Support/
        ├── AppSettings.swift        # backend URL + native-form-renderer toggle (UserDefaults)
        ├── SessionStore.swift       # persists session id/token (see its TODO on Keychain)
        ├── AnyCodable.swift         # type-erased box for dynamic JSON (answers/detail/result)
        └── ConditionExpression.swift # tiny visible_when/required_when evaluator (same-page only)
```

## Design notes

- **Registry dispatch** (`StepRegistry.swift`) is the whole point of the thin-shell architecture:
  `camera` → `CameraStepView`, `signature` → `SignatureStepView`, `pin` → `PinStepView` (these
  three are always rendered natively — the web renderer never implements them); `document` →
  always `WebStepView`; `form` → `WebStepView` unless the Settings toggle is on *and* the step
  only uses text/select/bool/date; `external` → `WebStepView` pointed at the vendor's own
  `webview_url`, with the session token deliberately **not** injected (only our own hybrid pages
  get the token).
- **Hybrid token handoff** (`WebStepView.swift`): the session token is injected into the WKWebView
  via a `WKUserScript` at document-start (`window.__SESSION_TOKEN__`), never through
  `localStorage`/cookies. A `WKScriptMessageHandler` named `bridge` is how the web content tells
  the shell a step finished — for our own pages, the shell just re-fetches `GET /sessions/{id}`;
  for external/vendor pages, the shell submits the generic `external` step body itself with
  whatever `result` the vendor page reported.
- **PIN is a credential, not an answer**: the value entered here authorizes the user rather than
  being a fact being collected, so it must never be treated like ordinary form input.
  `PinStepView` submits through the exact same `POST .../steps/{stepId}` call as every other step
  type — the server is responsible for handling it differently from a regular answer once it
  arrives — but the client-side discipline is just: never log it, never put it in the on-device
  draft store, clear the `@State` copy immediately after the call. A TODO marks where
  Keychain/Secure-Enclave storage for biometric unlock would go in a real build (out of POC
  scope).
- **On-device photo checks** (`CameraStepView.swift`): `VNDetectFaceRectanglesRequest` for
  face-present; the blur check is a stub that always passes — a real build would compute an
  actual sharpness metric, but the mock-capture images used in the simulator are flat-color
  placeholders that would always read as "blurry" under any real metric.
- **Progress bar / banners / repairs** are rendered by `RootView` around the registry's output,
  never inside a step renderer — they describe the state of the whole session (progress,
  system-wide banners, mid-flight repairs), not any single step, so they belong at the shell
  level, not inside a step's own view.
- **`force_update`** (the server's signal that the installed app version is no longer supported)
  is handled as a distinct root-level screen (`ForceUpdateView`), not routed through the registry
  at all, since it's a capability gate rather than a step.

## Known limitations (POC scope)

- **No real KMS/encryption** — nothing sensitive is stored client-side beyond a bearer token in
  `UserDefaults` (flagged with a TODO to move to Keychain in `SessionStore.swift`).
- **`device_attestation` is a hardcoded stub string** (`"poc-stub-attestation-token"`), not real
  App Attest.
- **No on-device draft store** for in-progress form input — `NativeFormStepView` is a stub for
  side-by-side comparison against the web renderer, not a production native renderer. A real
  build would need to persist in-progress answers locally so a user doesn't lose their input if
  the app is interrupted mid-form.
- **Blur detection in the camera step is a stub** (see above); face detection via Vision is real.
- **No automated tests** (unit or UI) — camera, signature, and PIN screens have no XCUITest
  coverage; they're exercised via manual simulator runs and code review against the step
  submission behavior defined in `docs/contract.md`.
- **iPhone-only** — `TARGETED_DEVICE_FAMILY` is set to `"1"` deliberately; no iPad or rotation
  support.
